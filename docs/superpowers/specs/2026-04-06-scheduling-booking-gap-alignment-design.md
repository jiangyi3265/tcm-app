# 排班/预约差异对齐设计

## 背景

仓库中已经实现了大部分医师排班、内部预约、公开预约、人员扩展资料、服务权限、实习生三天只读访问等能力，但与当前业务口径还有三个关键差异：

1. 非主治医生的近期病历访问窗口仍按 7 天计算，而不是 3 天。
2. 医师工作时间编辑虽然支持多时间段，但录入层没有强制半小时粒度。
3. 对外预约页已有“服务 + 医师 + 日期 + 时段”流程，但仍是列表式空闲时段选择，缺少基于工作时间和已预约信息的可视化周视图。

本轮不重做整套排班体系，只对现有实现做“规则对齐 + 交互升级”。

## 目标

1. 主治医生继续永久可见自己的病历；非主治临床角色的近期可见期统一收紧为 3 天。
2. 医师个人排班编辑和管理员排班编辑都只能录入半小时粒度时间。
3. 对外预约页支持按周查看工作时间、已占用时段和可预约空档。
4. 保持现有内部预约、自动匹配医师、服务权限、员工自动建档逻辑不回退。

## 范围

### 包含

- 前端 `permissions.js` 与后端 `PrivacyUtils.java` 的近期病历访问窗口对齐。
- 前端排班编辑器改为半小时粒度控件，并在保存前做前端校验。
- 后端工作时间规范化逻辑增加半小时粒度校验，拒绝非法时间段。
- 公开预约新增周视图排班/占用数据加载与渲染。
- 为上述改动补前后端测试。

### 不包含

- 不新增“按具体日期的例外停诊/加班”。
- 不重构内部预约页现有周视图实现。
- 不重做整个收银工作流，只保持现有授权服务限制不被破坏。

## 设计决策

### 1. 病历访问窗口

- 主治医生：永久可见自己的患者病历，维持现状。
- 实习生：继续按“实习日期当天起算共 3 天”的只读窗口，维持现状。
- 非主治 practitioner 类临床角色：将“近期诊疗可见期”从 7 天收紧到 3 天。
- pharmacist / cashier：保持现有按收费/处方状态衍生的访问规则，不被这次调整误伤。

这样可以满足“每个医生 3 天有效”的业务诉求，同时不破坏已上线的收银和药房权限逻辑。

### 2. 排班半小时粒度

前端两处排班编辑器都改成只能选择 `00:00` 到 `23:30`、步进 `00:30` 的时间控件。保存时增加以下校验：

- `start/end` 都必须存在。
- `start < end`。
- 分钟只能是 `00` 或 `30`。
- 同一天多个时间段按开始时间排序后不允许重叠。

后端继续以 `workingHours` 数组结构存储，但在 `normalizeWorkingHours` 里追加半小时粒度兜底校验，防止旧前端或手工请求写入非法时间。

### 3. 公开预约周视图

公开预约在现有列表模式上升级为“周视图时间表 + 选中槽位”交互：

- 顶部仍然先选服务、再选医师（可空）。
- 日期区改成一周导航，用户选择具体周后看到 7 天 x 半小时的网格。
- 选定医师时：
  - 工作时段显示为可视背景。
  - 已被预约的半小时块显示为占用。
  - 可预约起始槽显示为可点击态。
- 不选医师时：
  - 网格仅显示“至少有一名合资格医师在岗且可约”的空档。
  - 单元格附带默认分配医师和可匹配医师数量。

为避免公开端暴露完整内部预约详情，新增匿名周视图接口时只返回渲染所需的安全数据：

- 工作槽位状态：`off / working / booked / available`
- 时间标签和日期
- 聚合模式下的 `availablePractitionerIds / assignedPractitionerId / availableCount`

不返回病人姓名、电话或内部预约备注。

## 接口调整

### 后端新增匿名周视图接口

- `GET /api/public-booking/schedule`

请求参数：

- `serviceType`
- `practitionerId` 可空
- `weekStart`，格式 `YYYY-MM-DD`
- `roomId` 可选，供需要诊室的服务过滤

返回结构：

- `weekStart`
- `days[]`
- `slots[]`
  - `date`
  - `time`
  - `status`
  - `startTime`
  - `endTime`
  - `assignedPractitionerId`
  - `availablePractitionerIds`
  - `availableCount`

后端仍然复用现有 `workingHours + service duration + room conflict + appointment conflict + interval` 计算逻辑，只是把结果从“单日空闲起始时间列表”扩展为“一周半小时状态矩阵”。

## 前端落点

- `AppointmentView.vue`
  - 仅修正个人排班抽屉时间控件与序列化校验。
- `AdminView.vue`
  - 修正管理员编辑用户排班的时间控件与校验。
- `PublicBookingView.vue`
  - 从列表式时段选择升级为周视图网格。
  - 保留当前提交表单和 `assignedPractitionerId` 回填逻辑。
- `permissions.js`
  - 将非主治医生近期可见窗口从 7 天改为 3 天。

## 风险与规避

- 公开周视图会引入更多匿名查询：
  - 只返回脱敏状态矩阵，不返回预约详情。
- 半小时校验可能暴露旧脏数据：
  - 读取时仍兼容旧数据；只有保存新值时强制规范化。
- 前后端权限口径不一致会造成页面和接口判定不同步：
  - 本轮必须同时修改前端 `permissions.js` 与后端 `PrivacyUtils.java` 并补对应测试。

## 验证策略

- 前端权限测试：确认非主治医生 3 天内可见、超过 3 天不可见。
- 后端权限测试：与前端保持一致。
- 后端排班测试：拒绝 `09:10` / `09:45` 这类非法时间。
- 前端公开预约测试：确认周视图能正确展示工作、占用、可预约状态，并保持无医师场景下的自动匹配信息。
