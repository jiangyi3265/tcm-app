# 诊疗对比拷贝后内容闪退 Design

**日期：** 2026-04-06

## 目标
修复“诊疗记录 -> 对比往期 -> 全部拷贝到当前”时，文字先出现又被旧数据覆盖的问题。

## 现象
- 在已有诊疗记录上点“全部拷贝到当前”后，主诉/病程/辨证等文字会短暂显示。
- 若拷贝内容包含处方，界面随后会被旧诊疗数据覆盖，看起来像“没有赋值成功”。

## 根因
当前拷贝流程是：
1. 先把历史数据写入当前前端表单；
2. 若包含处方，再调用 `persistCopiedPrescriptions()`；
3. 对已有诊疗记录，`persistCopiedPrescriptions()` 不会先保存整张表单，而是直接走 `/api/consultations/{id}/prescriptions`；
4. 后端该接口只更新处方相关 payload，返回的仍是数据库里旧的 summary/diff/pricing；
5. 前端 `applySavedConsultation(updated)` 用整张旧数据覆盖表单，导致刚拷贝的文字消失。

## 方案对比
### 方案 A（采用）
- 新建记录：保持现有逻辑，先创建/保存，再同步处方。
- 已有记录：在前端完成整份拷贝后，直接保存整张诊疗单，让后端通过 `updateConsultation()` 一次性落库并联动库存。

**优点：**
- 改动最小；
- 数据源一致，不会出现“表单是新值、数据库还是旧值”；
- 已有后端 `updateConsultation()` 已具备处方库存联动兜底。

### 方案 B（不采用）
只改前端覆盖策略，让处方接口返回后不要整张覆盖表单。

**不采用原因：**
- 容易造成前后端状态不一致；
- 后续保存时仍可能被旧数据覆盖。

## 实施边界
- 只改前端诊疗拷贝流程；
- 不改 compare 面板 UI；
- 不改后端接口契约；
- 新增一个可单测的拷贝持久化流程辅助函数，便于回归验证。
